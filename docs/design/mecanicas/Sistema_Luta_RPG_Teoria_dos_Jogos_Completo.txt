Em vez de tratar o dano como mero cálculo numérico aleatório (ataque × multiplicador − defesa + ruído), podemos modelá-lo como o payoff central de um jogo estratégico entre agentes racionais (jogador e IA ou outro jogador). Isso transforma o combate de uma sequência de trocas de números em um espaço de decisões significativas, com profundidade emergente, risco calculado e múltiplas estratégias viáveis.
Abaixo, desenvolvo uma análise completa e aplicável, desde os fundamentos até implementações técnicas, com exemplos numéricos, pseudocódigo e considerações de design. O foco permanece no sistema de dano, mas sempre integrado ao ecossistema maior da batalha (recursos, cooldowns, elementos, status, targeting).
1. Modelando a Batalha por Turnos como Jogo Estratégico
Uma batalha por turnos é um jogo extensivo de forma finita (extensive-form game), frequentemente de soma zero (ou soma zero aproximada quando há empates por fuga ou mecânicas de sobrevivência). Os jogadores alternam ações em turnos discretos. O estado completo inclui:

HP (e outros recursos vitais) de todos os combatentes
MP/stamina, cooldowns, buffs/debuffs
Ordem de iniciativa (baseada em speed ou sistema de turnos)
Posição (em sistemas com grid ou front/back row)
Informações conhecidas vs. ocultas

Cada ação (ataque físico, habilidade elemental, defesa, uso de item, skill de status) é uma escolha estratégica. O sistema de dano determina a transição de estado: quanto HP é subtraído de quem, com que probabilidade, e com quais efeitos colaterais (crítico, status, knockback, etc.).
Payoff imediato = dano causado ao oponente − dano sofrido (ou, de forma mais sofisticada, variação na probabilidade de vitória a longo prazo). O objetivo terminal é reduzir o HP do(s) oponente(s) a zero antes que o seu chegue a zero (ou cumprir condição de vitória alternativa).
Essa modelagem permite aplicar diretamente conceitos como:

Equilíbrio de Nash
Minimax / Expectiminimax
Utilidade esperada
Jogos repetidos e indução backward
Informação perfeita vs. incompleta (Bayesian games)

2. O Sistema de Dano como Payoff e Fonte de Incerteza
O dano raramente é determinístico. Elementos típicos que introduzem estocasticidade:

Precisão (accuracy) vs. evasão
Faixa de dano (min–max) ou distribuição normal
Chance de crítico (crit chance + crit multiplier)
Multiplicadores elementais / de tipo (muitas vezes determinísticos, mas podem ser probabilísticos)
Efeitos de status com chance de aplicação
Resistências / fraquezas variáveis

Fórmula genérica de dano esperado (E[D]):
$$E[D] = p_{\text{acerto}} \times \Bigl( D_{\text{base}} + p_{\text{crit}} \times (D_{\text{crit multiplier}} - 1) \times D_{\text{base}} \Bigr) \times M_{\text{elemental}} \times (1 + \text{variance factor})$$
Onde $  M_{\text{elemental}}  $ pode ser 2.0 (fraqueza), 0.5 (resistência) ou 1.0 (neutro). A variância real (rolls de dado ou RNG) exige que a IA/jogador raciocine em termos de utilidade esperada, não de dano médio isolado.
Aqui entra a teoria dos jogos: o dano não é só “número maior = melhor”. Ele é o resultado de uma escolha estratégica que depende do que o oponente provavelmente fará no mesmo turno ou nos seguintes.
3. Estratégias Puras, Mistas e Equilíbrio de Nash no Contexto de Dano
Quando o sistema de dano cria ciclo de counter (ex.: Fogo > Gelo > Terra > Fogo), temos um jogo de soma zero finito que se assemelha a Pedra-Papel-Tesoura (RPS). Em RPS clássico, não existe estratégia pura dominante; o equilíbrio de Nash é a estratégia mista uniforme (1/3 para cada).
Exemplo numérico simples (1v1, duas ações cada)
Suponha Guerreiro vs. Mago elemental. Ações:

Guerreiro: Slash (dano físico esperado 60, 90% acc) ou Fire Strike (dano fogo esperado 55, 75% acc)
Mago: Ice Bolt (dano gelo esperado 50) ou Earth Spike (dano terra esperado 65)

Matriz de payoff (valor = dano esperado causado pelo Guerreiro − dano esperado sofrido; números hipotéticos calculados com o sistema de dano do jogo):




















Ação do Guerreiro \ Ação do MagoIce BoltEarth SpikeSlash (físico)+35−10Fire Strike (fogo)−5+25
Não há saddle point (valor que é maxmin = minmax). O Guerreiro quer maximizar o mínimo; o Mago quer minimizar o máximo.
Para encontrar o equilíbrio de Nash misto:
Seja $  p  $ = probabilidade de o Guerreiro escolher Slash.
O Mago fica indiferente quando o payoff esperado das duas colunas é igual:
$$p \times 35 + (1-p) \times (-5) = p \times (-10) + (1-p) \times 25$$
Resolvendo:
$$35p - 5 + 5p = -10p + 25 - 25p$$
$$40p + 5 = 15 - 35p$$
$$75p = 10 \implies p = \frac{2}{15} \approx 0.133$$
Ou seja, o Guerreiro deve escolher Slash apenas ~13% das vezes e Fire Strike ~87% para tornar o Mago indiferente. O valor do jogo (expected payoff no equilíbrio) pode ser calculado substituindo de volta. O Mago ajusta sua própria probabilidade de forma análoga.
Implicação para design de dano: Sistemas elementais puros demais criam RPS rígido. Para torná-lo mais rico, adicione ruído (accuracy diferente, custos de MP distintos, efeitos secundários como burn/poison que persistem) ou camadas (status que alteram a matriz de payoff dinamicamente). O artigo de Denis Osipov sobre Game Theory in Combat Design enfatiza exatamente isso: usar matrizes de estratégia para diagnosticar estratégias dominantes e restaurar diversidade de escolhas.⁠LinkedIn
4. Expectiminimax: O Algoritmo Ideal para Sistemas de Dano Estocástico
Para IA (ou para simular o “melhor play” humano), o algoritmo clássico minimax assume jogo determinístico. Com dano aleatório, precisamos de Expectiminimax (ou Expectimax quando há apenas um jogador maximizando contra a natureza).⁠Wikipedia
Estrutura da árvore:

Nós Max: turno do agente que queremos otimizar (escolhe ação que maximiza o valor esperado)
Nós Min: turno do oponente (escolhe ação que minimiza o valor do agente)
Nós Chance: após uma ação, rola o dano (ou acerto/crit). O valor é a média ponderada pelas probabilidades:

$$V_{\text{chance}} = \sum_i p_i \times V(\text{estado}_i)$$
Pseudocódigo adaptado (profundidade limitada + heurística de avaliação de estado):

Heurística de avaliação deve ser sofisticada: não apenas diferença de HP, mas probabilidade estimada de vitória considerando recursos restantes, ameaças de burst, etc. Técnicas modernas combinam Expectiminimax com Monte Carlo Tree Search (MCTS) para lidar com branching factor alto (muitas skills × muitos alvos × variância de dano).
Em sistemas com dano muito variável, o Expectiminimax faz a IA preferir consistentemente ações de “bom dano esperado + baixa variância” quando está à frente, ou “high risk/high reward” quando precisa de comeback — exatamente o raciocínio que um jogador humano experiente faria.
5. Informação Incompleta, Bayesian Updating e Adaptação
Em muitos RPGs, o jogador não conhece todos os stats, fraquezas ou cooldowns do inimigo (especialmente bosses ou inimigos novos). Isso transforma o jogo em um jogo bayesiano.
O agente mantém uma distribuição de crença sobre o “tipo” do oponente (ex.: “70% chance de ser fraco a Fogo, 20% neutro, 10% resistente”). Cada ação observada (o inimigo usou uma skill de gelo repetidamente) atualiza as probabilidades via teorema de Bayes. O dano esperado futuro é então calculado sob a crença atualizada.
Isso adiciona profundidade enorme ao sistema de dano: testar fraquezas elementais vira uma decisão estratégica de “vale a pena arriscar um turno para descobrir?” vs. “usar o que já sei que funciona”.
6. Jogos Repetidos, Cooldowns e Gestão de Recursos
Batalhas longas são jogos repetidos (ou extensivos com horizonte finito). Usar uma habilidade de alto dano com cooldown de 3 turnos tem custo de oportunidade alto. A teoria recomenda indução backward: resolver primeiro os últimos turnos e voltar, considerando o valor de poupar recursos.
Exemplo clássico: “Devo usar meu ultimate agora (dano massivo esperado 300) ou guardar para quando o boss entrar em enrage?” A resposta depende do estado atual de HP, da probabilidade de o oponente permitir o setup, e do valor de negar um turno via stun vs. dano bruto.
Sistemas de dano que ignoram isso (dano flat por turno sem trade-offs) tendem a ser menos interessantes.
7. Balanceamento e Testes Usando Ferramentas da Teoria dos Jogos
Como designer:

Construa matrizes de matchup para archetypes de builds ou para pares de habilidades chave.
Simule milhares de batalhas com agentes que jogam segundo diferentes estratégias (puras, mistas, heurísticas simples vs. Expectiminimax profundo).
Meça win rates, tempo médio de batalha, variedade de ações usadas e “dominância” (uma estratégia vence >70-80% das vezes?).
Ajuste multiplicadores de dano, accuracy, custos e efeitos secundários até que o equilíbrio se aproxime de Nash (múltiplas estratégias com win rate próximo de 50% contra o meta).

Ferramentas como evolutionary game theory (populações de builds “evoluem” via seleção) ajudam a descobrir estratégias inesperadas que humanos não testariam manualmente.
8. Considerações Práticas de Implementação e Limitações
Complexidade computacional: Branching factor explode com party de 4 + múltiplos inimigos + muitas skills. Soluções:

Limitar profundidade + boa heurística
MCTS com rollout usando simulações rápidas de dano
Agrupar ações similares ou usar aprendizado por reforço (RL) para aproximar a política ótima

Diversão vs. Otimalidade: Uma IA perfeita de Expectiminimax pode ser frustrante se o jogador humano comete erros. Soluções comuns: dificuldade dinâmica (a IA “erra” mais em níveis baixos) ou dar ao jogador ferramentas de informação/outplay (previsão de padrões, counters explícitos).
Clareza e Determinismo: O sistema precisa comunicar claramente as consequências esperadas das ações (tooltips com “dano esperado: 45-72”, “chance de aplicar burn: 40%”). Sem clareza, o jogador não consegue formar estratégias racionais.
Emergent complexity a partir de regras simples: Como destacado em análises de design de turn-based combat, sistemas com poucas regras bem interligadas (dano + elementos + status + recursos limitados) geram mais profundidade do que tabelas gigantes de números.⁠Sinisterdesign
Conclusão e Extensões
Aplicar teoria dos jogos ao sistema de dano não significa tornar o jogo “mais matemático” para o jogador final — significa projetar mecânicas onde as escolhas pareçam matemáticas profundas e recompensadoras. O dano deixa de ser um botão de “mais dano = melhor” e passa a ser uma alavanca em um sistema de payoffs interdependentes.
Extensões modernas incluem:

Combinação com aprendizado por reforço (a política do agente é treinada para maximizar utilidade esperada em ambientes com dano estocástico)
Análise de equilíbrio em jogos de múltiplos jogadores (party vs. múltiplos bosses ou PvP)
Mecânicas de incomplete information intencionais (fog of war em stats ou “descoberta” de fraquezas)

O resultado é combate com profundidade estratégica real, replayability alta e sensação de maestria — exatamente o que diferencia sistemas de dano memoráveis (Press Turn de Shin Megami Tensei, triangle weapons de Fire Emblem, complexidade de status em Persona) daqueles que se reduzem a “apertar Attack até morrer”.

2. Como o Dano Estocástico se Integra com a Teoria dos Jogos
Toda vez que apply_action é chamada, o dano é amostrado (não esperado teórico). Isso cria implicitamente os chance nodes do Expectiminimax. Durante os milhares de rollouts do MCTS, a média dos resultados converge para o valor esperado real do estado, exatamente como a fórmula:
$$V_{\text{chance}} = \sum p_i \cdot V(\text{estado após outcome}_i)$$
No rollout usamos amostragem de Monte Carlo; na árvore usamos UCT para focar nas ações mais promissoras. O resultado é uma IA que:

Prefere ações com bom dano esperado + consistência quando está à frente.
Aceita risco calculado (skills de maior variância) quando precisa de comeback.
Evita ações com alta chance de miss em momentos críticos.

5. Como Estender para um RPG Real
A. Party completa (4xN)

Adicione uma lista de personagens + fila de turnos baseada em speed.
get_legal_actions passa a incluir escolha de alvo + skill.
O estado cresce, mas MCTS com 800–2000 iterações ainda funciona bem em turn-based.

B. Chance nodes explícitos (mais preciso que puro sampling)
Quando os outcomes são discretos e poucos (ex: Miss / Hit normal / Crítico), crie filhos de chance no expand com pesos = probabilidades. No backpropagation use média ponderada em vez de amostra única. Isso é Expectiminimax “puro” dentro do MCTS.
C. Heurística mais forte
Substitua _evaluate_state por uma rede neural leve treinada (ou mesmo uma função com mais features: ameaças de burst, cooldowns restantes, sinergia de status). MCTS + rede = AlphaZero-style para RPG.
D. Integração com engines

Godot (GDScript/Python): exponha a classe ExpectiMCTS via GDExtension ou chame Python externo.
Unity (C#): portar a lógica é direto (dataclasses → classes, random → UnityEngine.Random).
Qt6/C++: use o Python como protótipo/ferramenta de balanceamento e depois implemente versão otimizada com std::mt19937 e std::vector.

E. Tuning de parâmetros

iterations: 400–1200 por turno (testar em seu hardware).
exploration_weight (C): 1.2–1.8 (maior = mais exploração).
rollout_depth: 8–15 (muito profundo = lento; muito raso = míope).
Política de rollout: quanto mais gulosa em dano esperado, mais forte a IA (mas menos exploração de linhas arriscadas).

6. Considerações de Design e Balanceamento
Esta implementação revela rapidamente problemas no seu sistema de dano:

Se o MCTS sempre escolhe a mesma ação → você tem estratégia dominante (ajuste multiplicadores elementais, custos de MP ou adicione counters).
Se a IA perde muito para o inimigo guloso → o sistema de dano está desbalanceado (ex: skills muito fortes ou defesa irrelevante).
Teste com populações de builds diferentes (evolutionary game theory) para encontrar equilíbrios de Nash aproximados.

O código já inclui logs detalhados de cada dano — excelente para depurar e ajustar a fórmula de dano (accuracy, k da defesa, chance de status, etc.).

1. Modelagem atual como jogo extensivo assimétrico
A batalha é um jogo extensivo de forma finita com condição terminal assimétrica:

Jogador maximizador: escolhe (a) qual membro da party age, (b) qual habilidade/item, (c) qual inimigo mirar. Objetivo primário = sobreviver com Gus vivo + reduzir HP inimigo a zero.
Inimigos minimizadores: tentam reduzir HP de Gus a zero o mais rápido possível.
Payoff terminal:
Gus morre ou foge → utilidade extremamente negativa para o jogador (ex: -10.000 ou perda imediata).
Inimigos todos mortos → +1.000 + bônus por recursos restantes.
Fuga bem-sucedida → utilidade intermediária (depende do contexto da história).


Essa assimetria já é excelente do ponto de vista de teoria dos jogos: o jogador é forçado a tomar decisões de proteção de ativo crítico (Gus é como o “rei” ou o “payload” em um jogo de escort). Buffs defensivos em Gus, debuffs em inimigos que ameaçam Gus, e escolha de quem ataca primeiro viram decisões de alto valor.
O problema atual é que o lado inimigo tem uma estratégia pura dominante (“sempre ataque Gus”). Em teoria dos jogos, estratégias puras dominantes reduzem a profundidade e tornam o jogo menos interessante (o jogador aprende rápido e otimiza em torno disso).
2. Sugestões concretas para tornar o targeting inimigo menos determinístico (mas ainda inteligente)
Aqui vão as melhores abordagens, ordenadas da mais recomendada para a mais avançada:
A. Sistema de Threat / Aggro Score + Seleção Probabilística (melhor equilíbrio)
Cada inimigo calcula um score de ameaça para cada membro da party no início do seu turno. Gus começa com score base muito alto (ex: 100), mas outros membros podem subir temporariamente.
Fórmula de Threat Score (exemplo para um inimigo):
$$\text{Threat}_i = w_1 \cdot \text{BaseThreat}_i + w_2 \cdot \text{DanoCausadoRecentemente}_i + w_3 \cdot \text{BuffsVisíveisEm}_i - w_4 \cdot \text{DebuffsEm}_i + w_5 \cdot \text{HPInverso}_i$$

BaseThreat_Gus = 100 (ou 80–120 dependendo da “personalidade” do inimigo)
BaseThreat_outros = 10–30
DanoCausadoRecentemente: se o membro X deu muito dano no último turno, seu threat sobe (“ele é perigoso”).
BuffsVisíveisEm: se Gus está com buff de defesa forte visível, threat dele pode cair ligeiramente (inimigo “percebe” que não vale tanto a pena).
HPInverso: membros com HP baixo podem atrair mais atenção (“alvo fraco”).

Depois de calcular os scores, o inimigo não escolhe sempre o maior. Em vez disso:
Opção 1 – Epsilon-greedy (mais simples e garantida)
PythonCopiarepsilon = 0.08  # 8% de chance de atacar alguém que não é o melhor
if random.random() < epsilon:
    target = random.choice(party_members_except_best)
else:
    target = argmax(Threat_scores)
Isso garante mínimo de 8% de chance de atacar outros membros, independentemente do estado. Fácil de balancear e explicar ao jogador (“às vezes o inimigo se distrai ou erra de alvo”).
Opção 2 – Softmax / Boltzmann (mais elegante e “game-theoretic”)
Converte os threat scores em probabilidades de forma suave:
$$P(\text{atacar } i) = \frac{\exp(\text{Threat}_i / T)}{\sum_j \exp(\text{Threat}_j / T)}$$
Onde $  T  $ é a temperatura:

$  T = 8 \sim 12  $: bastante aleatório (bom para inimigos “bestas” ou em pânico).
$  T = 3 \sim 5  $: ainda prioriza Gus fortemente, mas dá chance realista de mirar em outros (recomendado).
$  T \to 0  $: volta a ser quase determinístico (use para bosses muito focados).

Essa é basicamente uma estratégia mista aproximada. O inimigo ainda joga “quase otimamente”, mas com ruído racional.
B. Sticky Target + Threshold
O inimigo lembra quem atacou por último. Para mudar de alvo, o novo candidato precisa superar o threat do alvo atual em pelo menos X% (ex: 25%). Isso evita que o targeting fique pulando toda hora e dá sensação de “personalidade”.
C. Integração com o ExpectiMCTS que já implementamos
No código atual, o inimigo usa uma heurística gulosa de dano esperado. Podemos melhorar:

No método _expected_damage_heuristic ou ao escolher ação do inimigo, primeiro calcule best_target = argmax(Threat) usando o sistema acima.
Em vez de sempre usar best_target, amostre o alvo usando softmax ou epsilon-greedy antes de calcular o dano.
Dentro do MCTS, quando for nó Min (turno inimigo), ao expandir ações, expanda também as variações de alvo com suas probabilidades. O valor backpropagado já considera que Gus morrer = utilidade extremamente baixa.

Isso faz o MCTS “entender” que, mesmo com targeting probabilístico, proteger Gus continua sendo a prioridade número 1.
D. Personalidades de Inimigo (evolutionary / repeated game)
Diferentes tipos de inimigo têm vetores de pesos diferentes nos Threat Scores:

“Bruto”: alto peso em dano recente + HP baixo (ataca quem está machucando ele).
“Focado em Gus”: base Gus muito alta + ignora quase todo resto.
“Oportunista”: sobe threat de quem está com debuff forte ou HP baixo.
“Enraivecido” (quando próprio HP < 30%): aumenta drasticamente o peso de Gus.

Isso transforma a IA inimiga em um jogo repetido com memória e adaptação — muito mais interessante do ponto de vista teórico e de jogabilidade.
3. Outras aplicações de teoria dos jogos no sistema completo
Recursos (HP + Mana) como jogo repetido finito
Cada turno é uma etapa de um jogo repetido com horizonte limitado. Usar mana agora para um buff forte em Gus tem custo de oportunidade alto (pode faltar mana para cura depois). O Expectiminimax/MCTS já captura isso naturalmente via rollouts profundos. Sugestão: adicione um termo de “custo de oportunidade futuro” na heurística de avaliação.
Escolha de itens de buff/debuff
É um sub-jogo de alocação de recursos. O jogador decide quem recebe o buff (Gus recebe defensive buff? Ou o DPS recebe offensive buff para matar a ameaça mais rápido?). Inimigos podem ter ações de dispell ou steal de buff — isso vira um mini-jogo de informação e contra-jogo.
Fuga (Escape)
Modelar como uma ação arriscada com probabilidade de sucesso dependente de:

Velocidade média da party vs inimigos
Itens de fuga
Estado de Gus (se Gus está com HP baixo, chance de fuga cai)

Utilidade da fuga = valor esperado de sobreviver com Gus vivo menos penalidade (perder itens, moral da party, etc.). Em teoria dos jogos, é uma opção de “sair do jogo” com payoff intermediário.
Win condition “Gus morre = party lose”
Isso é similar a jogos de “assassinate the VIP” ou “king of the hill”. O valor de qualquer dano em Gus deve ser multiplicado por um fator muito alto na função de avaliação do MCTS. Exemplo:
$$\text{Utility} = (\text{HP\_Gus\_normalizado} \times 5.0) + (\text{HP\_outros\_normalizado} \times 1.0) - (\text{HP\_inimigos} \times 0.8)$$
Dessa forma o MCTS (e a IA inimiga) naturalmente prioriza Gus sem precisar de regras hardcoded.

Camada 2 – Sistema de Threat / Aggro Avançado + Estratégia Mista (o coração da sua pergunta)
Esta é a evolução direta do que discutimos antes. Objetivo: inimigos priorizam Gus de forma racional, mas com chance mínima e controlada de mirar em outros.
Threat Score Multifatorial (exemplo completo)
PythonCopiardef calculate_threat(self, enemy: CharacterState, target: CharacterState, state: GameState) -> float:
    threat = 0.0

    # 1. Base (Gus tem peso enorme)
    if target.is_gus:
        threat += 120.0
    else:
        threat += 18.0

    # 2. Dano recente causado por este alvo (memória curta)
    threat += target.last_damage_dealt * 2.8

    # 3. HP inverso (alvos feridos são mais atrativos para finalizar)
    threat += (1.0 - target.hp / target.max_hp) * 35.0

    # 4. Buffs/Debuffs visíveis (informação imperfeita parcial)
    for status in target.statuses:
        if status.type in ["AttackUp", "CritUp"]:
            threat += 22.0
        if status.type in ["DefenseDown", "Vulnerable"]:
            threat += 28.0
        if status.type == "Taunt" and status.source == enemy.id:
            threat += 80.0   # forte resposta ao taunt

    # 5. Ameaça acumulada (decai a cada turno global)
    threat += target.threat_generated * 0.6

    # 6. Fator de “distância” ou formação (se você adicionar grid)
    # threat += distance_penalty(enemy.pos, target.pos) * 15

    # 7. Personalidade do inimigo (vetor de pesos)
    if enemy.personality == "focused_gus":
        threat *= 1.35 if target.is_gus else 0.6
    elif enemy.personality == "opportunist":
        threat += (1.0 - target.hp / target.max_hp) * 45

    return max(1.0, threat)
Seleção Probabilística (duas opções fortes)
A. Softmax com Temperatura Dinâmica (recomendado)
PythonCopiardef choose_target_mixed(self, enemy, possible_targets, state):
    threats = [self.calculate_threat(enemy, t, state) for t in possible_targets]
    T = 4.2   # temperatura base (ajuste por tipo de inimigo)
    
    # Temperatura sobe se inimigo está em pânico (HP baixo)
    if enemy.hp / enemy.max_hp < 0.35:
        T *= 1.6
    
    exp_threats = [math.exp(th / T) for th in threats]
    total = sum(exp_threats)
    probs = [e / total for e in exp_threats]
    
    return random.choices(possible_targets, weights=probs, k=1)[0]

7% → segundo melhor

PythonCopiardef evaluate_state(state: GameState) -> float:
    if state.is_terminal():
        if any(e.hp <= 0 for e in state.enemies if all dead):
            return 1200 + (state.party_gus.hp / state.party_gus.max_hp) * 400
        if state.party_gus.hp <= 0 or fled:
            return -15000   # morte de Gus é catastrófica
        return 0

    gus = state.get_gus()
    score = 0.0

    # 1. Sobrevivência de Gus (peso dominante)
    gus_hp_norm = gus.hp / gus.max_hp
    score += gus_hp_norm * 650.0
    if gus_hp_norm < 0.25:
        score -= 180   # pânico

    # 2. HP dos outros membros (peso menor)
    for member in state.party:
        if not member.is_gus:
            score += (member.hp / member.max_hp) * 85.0

    # 3. Recursos (mana + cooldowns)
    score += (gus.mp / gus.max_mp) * 45
    for member in state.party:
        score += sum(1 for cd in member.cooldowns.values() if cd == 0) * 8

    # 4. Ameaça inimiga (priorizar matar os que mais ameaçam Gus)
    for enemy in state.enemies:
        threat_to_gus = self.calculate_threat(enemy, gus, state)
        score -= threat_to_gus * 1.8
        score -= (enemy.hp / enemy.max_hp) * 40   # ainda queremos matar

    # 5. Status effects (sinergia)
    for status in gus.statuses:
        if "Defense" in status.type or "Regen" in status.type:
            score += 35
        if "Taunt" in status.type:
            score += 22   # bom se estiver funcionando

    # 6. Economia de turnos / setup
    if state.turn_count > 12:
        score -= (state.turn_count - 12) * 8   # pressão de tempo

    return score + random.uniform(-12, 12)   # ruído pequeno para robustez
